# Trindex Memory System Improvement Plan

**Date:** 2026-03-07  
**Context:** Principal Engineering Assessment of Memory System Architecture  
**Scope:** Problems 1-6 with 3 solution options each

---

## Problem 1: Deduplication on Write

**Current State:** Agents store memories without checking for duplicates, resulting in multiple identical or semantically equivalent entries (e.g., 5+ copies of "User's name is Dave").

**Impact:** Storage bloat, retrieval noise, confusion about which memory is authoritative.

### Option A: Client-Side Semantic Check (Recommended)

**Approach:** Before calling `remember`, agent queries with high similarity threshold (≥0.95). If match found within namespace, skip storage and optionally update existing.

**Implementation:**
```go
// Pseudo-code for remember_unique
func RememberUnique(content, namespace string, threshold float64) error {
    existing := Recall(content, namespace, threshold, 1)
    if len(existing) > 0 && existing[0].Score >= threshold {
        // Optional: Update timestamp or metadata
        return nil // Skip duplicate
    }
    return Remember(content, namespace)
}
```

**Pros:**
- No server changes required
- Immediate deployment across all agents
- Agents control dedup strictness per use case

**Cons:**
- Extra round-trip (recall + remember)
- Race condition window between check and write
- Inconsistent if agents don't all implement

**Effort:** Low (1-2 days)

### Option B: Server-Side Hash-Based Deduplication

**Approach:** Store content hash (SHA-256) as indexed column. On insert, reject if hash exists within namespace. Fall back to semantic check for near-duplicates.

**Implementation:**
- Add `content_hash` column to memories table
- Create unique constraint: `(namespace, content_hash)`
- Return existing ID on conflict instead of error

**Pros:**
- Zero latency overhead (hash check is O(1))
- Guaranteed deduplication at storage layer
- No race conditions

**Cons:**
- Requires schema migration
- Identical content with different formatting creates new entry
- Doesn't catch semantic duplicates ("Dave" vs "User is Dave")

**Effort:** Medium (3-5 days including migration)

### Option C: Vector Index Deduplication with Merge Strategy

**Approach:** Use existing pgvector index to find near-duplicates at write time. If similarity > threshold, merge content (concatenate or pick longer) rather than reject.

**Implementation:**
- Modify `remember` to run vector similarity search before insert
- On match: `new_content = merge(existing.content, new_content)`
- Update existing row instead of inserting new

**Pros:**
- Catches semantic duplicates, not just exact matches
- Content enrichment over time (details accumulate)
- Single operation (no extra client round-trip)

**Cons:**
- Complex merge logic (what constitutes "better" content?)
- Write latency increases (vector search on every remember)
- Potential for runaway content growth

**Effort:** High (1-2 weeks, complex merge heuristics)

**Recommendation:** Start with Option A (client-side) for immediate relief. Implement Option B (hash-based) as server-side safety net. Defer Option C until we understand merge patterns better.

## Problem 2: Namespace Strategy

**Current State:** Ad-hoc namespaces (`default`, `user_info`, `trindex`). No clear ownership or conventions. Agents dump memories wherever, making retrieval unreliable.

**Impact:** Cross-agent context loss, inability to scope searches, namespace pollution.

### Option A: Hierarchical Namespace Convention

**Approach:** Establish strict hierarchy: `global` > `project:{name}` > `agent:{name}` > `session:{id}`. Agents always write to most specific appropriate level.

**Rules:**
- `global`: User identity, preferences, cross-project facts
- `project:{name}`: Architecture, decisions, codebase patterns
- `agent:{name}`: Agent-specific learnings (rarely needed)
- `session:{id}`: Ephemeral context (auto-expires)

**Implementation:**
- Document convention in CLAUDE.md
- Add validation on `remember` (reject unknown namespaces?)
- `recall` automatically searches up hierarchy

**Pros:**
- Clear mental model
- Automatic scoping (session → project → global)
- Prevents pollution

**Cons:**
- Requires discipline (easy to misuse)
- Migration pain for existing memories
- Hierarchy depth may not fit all use cases

**Effort:** Low (documentation + convention enforcement)

### Option B: Tag-Based Flat Namespaces

**Approach:** Flat namespace structure with rich tagging. Query by tag intersection rather than namespace hierarchy.

**Structure:**
```json
{
  "namespace": "trindex",
  "tags": ["user:dave", "project:trindex", "type:architecture", "agent:claude"]
}
```

**Implementation:**
- Replace namespace with tags array
- Index tags for fast filtering
- Queries specify tag sets: `recall(query, tags=["user:dave", "type:bug"])`

**Pros:**
- Flexible cross-cutting concerns
- No migration (tags can be added to existing)
- Multiple valid categorizations per memory

**Cons:**
- Tag proliferation (agents inventing own tags)
- Query complexity (AND vs OR semantics)
- Harder to reason about "where" to put things

**Effort:** Medium (schema change + query builder)

### Option C: Automatic Namespace Detection

**Approach:** AI-powered namespace classification. Agent provides raw content, system determines optimal namespace based on content analysis.

**Implementation:**
```go
func AutoNamespace(content string) string {
    // Use lightweight classifier or LLM call
    if contains(content, ["user", "name", "preference"]) {
        return "global"
    }
    if contains(content, ["project", "architecture", "decision"]) {
        return "project:" + detectProject()
    }
    return "default"
}
```

**Pros:**
- Zero cognitive load on agents
- Consistent categorization
- Adapts to new namespace patterns

**Cons:**
- Classification errors (miscategorized memories)
- Latency (LLM call or embedding lookup)
- Opacity (why was this put here?)

**Effort:** High (ML pipeline or LLM integration)

**Recommendation:** Option A (hierarchical) for immediate clarity. Add tagging (Option B elements) as metadata enrichment later. Avoid Option C until classification is highly reliable.

## Problem 3: Structured Metadata

**Current State:** Memories are plain text strings. No type information, source tracking, or confidence scoring. Impossible to filter ("show me only bugs") or validate.

**Impact:** Retrieval returns irrelevant results, no provenance, hard to audit.

### Option A: JSON Metadata Column

**Approach:** Add `metadata` JSONB column to memories table. Store structured fields alongside content. Query using JSON operators.

**Schema:**
```sql
ALTER TABLE memories ADD COLUMN metadata JSONB DEFAULT '{}';
CREATE INDEX idx_metadata_type ON memories ((metadata->>'type'));
```

**Example:**
```json
{
  "content": "User prefers dark mode",
  "metadata": {
    "type": "preference",
    "source": "user-statement",
    "confidence": 0.95,
    "agent": "claude",
    "created_at": "2026-03-07T03:00:00Z"
  }
}
```

**Pros:**
- Flexible schema (evolves without migrations)
- Native JSON operators in PostgreSQL
- Backwards compatible (default empty object)

**Cons:**
- No validation (agents can store any structure)
- Query performance on deeply nested fields
- Storage overhead (JSON parsing)

**Effort:** Low (1 day + index tuning)

### Option B: Typed Memory System

**Approach:** Define memory types as enums with validated schemas. Each type has required fields. Store in separate tables or with discriminator.

**Types:**
```go
type MemoryType string
const (
    MemoryFact        MemoryType = "fact"
    MemoryPreference  MemoryType = "preference"
    MemoryDecision    MemoryType = "decision"
    MemoryBug         MemoryType = "bug"
    MemoryPattern     MemoryType = "pattern"
)

type TypedMemory struct {
    ID        uuid.UUID
    Type      MemoryType
    Content   string
    // Type-specific fields
    Facts     *FactMetadata     // when Type == "fact"
    Bugs      *BugMetadata      // when Type == "bug"
    // ...
}
```

**Pros:**
- Type safety (compile-time validation)
- Optimized storage per type
- Clear API contracts

**Cons:**
- Schema rigidity (new types require migration)
- Complex query logic (joins or unions)
- Breaking change for existing memories

**Effort:** High (2-3 weeks, significant refactor)

### Option C: Content-Derived Metadata Extraction

**Approach:** Automatically extract metadata from content using NLP/LLM. No explicit metadata from agents. System parses content for type, confidence, relationships.

**Implementation:**
```go
func ExtractMetadata(content string) Metadata {
    // Use NER, classification, or lightweight LLM
    return Metadata{
        Type:      classify(content), // "fact", "preference", etc.
        Entities:  extractEntities(content), // ["Dave", "Trindex"]
        Sentiment: analyzeSentiment(content),
        Confidence: 0.8, // based on certainty markers
    }
}
```

**Pros:**
- Zero agent changes
- Rich metadata without discipline
- Can retroactively process existing memories

**Cons:**
- Extraction errors
- Latency on write (or async processing complexity)
- Opacity (why was this classified as "preference"?)

**Effort:** Medium-High (ML pipeline + validation)

**Recommendation:** Option A (JSON metadata) for flexibility. Define recommended schema in documentation but don't enforce rigidly. Build validation layer later if needed. Defer Option B until we know exactly what fields we need.

## Problem 4: Retrieval Hygiene

**Current State:** Agents may not query at session start, or query with inappropriate thresholds. Low thresholds (0.0001) return noise; high thresholds miss relevant context.

**Impact:** Context loss between sessions, retrieval returning garbage or missing key facts.

### Option A: Mandatory Session Initialization Pattern

**Approach:** Define standard "session bootstrap" that agents MUST run at start. Queries `global` + relevant project namespaces, caches results for session.

**Pattern:**
```go
func InitializeSession(agentName, projectName string) SessionContext {
    return SessionContext{
        GlobalFacts:     Recall("user identity preferences", ["global"], 0.3, 10),
        ProjectFacts:    Recall("project context", [projectName], 0.3, 10),
        RecentActivity:  Recall("recent work", [projectName], 0.1, 5, since=24h),
    }
}
```

**Enforcement:**
- Document in CLAUDE.md as REQUIRED
- Add linter/check for agent code (look for InitializeSession call)
- Framework-level: Auto-inject context if agent uses Trindex MCP

**Pros:**
- Guaranteed context loading
- Consistent thresholds per query type
- Predictable agent behavior

**Cons:**
- Additional latency at session start
- One-size-fits-all may not fit all use cases
- Hard to enforce across different agent implementations

**Effort:** Low (documentation + code review)

### Option B: Adaptive Threshold System

**Approach:** Dynamic thresholds based on query characteristics. Broad queries get lower thresholds; specific queries get higher. System learns optimal thresholds per namespace.

**Rules:**
```go
func GetThreshold(query string, namespace string) float64 {
    if isSpecific(query) { // contains proper nouns, unique terms
        return 0.7
    }
    if namespace == "global" {
        return 0.5 // user facts need precision
    }
    return 0.3 // project context can be broader
}
```

**Machine Learning:** Track which memories were actually useful (agent upvotes/downvotes), adjust thresholds per query pattern.

**Pros:**
- Optimal retrieval without manual tuning
- Adapts to different content types
- Better precision/recall tradeoff

**Cons:**
- Complex to implement and tune
- Requires feedback loop (agents reporting usefulness)
- Unpredictable for debugging

**Effort:** High (ML system + feedback mechanism)

### Option C: Context Window Management

**Approach:** Treat memories like LLM context window. Rank by relevance + recency, include top N in prompt context. Automatic pruning of low-value memories.

**Implementation:**
```go
func BuildContextWindow(query string, maxTokens int) []Memory {
    candidates := Recall(query, allNamespaces, 0.1, 100)
    ranked := rankBy(candidates, 
        score*0.5 +     // relevance
        recency*0.3 +   // recent = more important
        typeBoost*0.2   // preferences > random facts
    )
    return fitToTokenBudget(ranked, maxTokens)
}
```

**Pros:**
- Automatic relevance filtering
- Fits within LLM context limits
- Surfaces most important context

**Cons:**
- May drop relevant but older memories
- Complex ranking heuristics
- Non-deterministic (hard to reproduce)

**Effort:** Medium (ranking algorithm + token counting)

**Recommendation:** Option A (mandatory initialization) immediately. It's documentation + discipline, no code changes. Experiment with Option C (context window) for complex multi-turn sessions. Avoid Option B until we have usage data.

## Problem 5: Memory Lifecycle

**Current State:** All memories persist forever. No expiration for ephemeral context (session IDs, temporary files, transient errors). Storage grows indefinitely.

**Impact:** Cluttered retrieval, storage costs, old irrelevant memories drowning out current context.

### Option A: TTL (Time-To-Live) with Auto-Expiration

**Approach:** Add `expires_at` timestamp to memories. Background job deletes expired entries. Agents can specify TTL per memory.

**Implementation:**
```sql
ALTER TABLE memories ADD COLUMN expires_at TIMESTAMP;
CREATE INDEX idx_expires_at ON memories (expires_at) WHERE expires_at IS NOT NULL;

-- Background job runs every hour
DELETE FROM memories WHERE expires_at < NOW();
```

**Usage:**
```go
RememberWithTTL("Session context: debugging auth", "session:abc123", 24*time.Hour)
```

**Default TTLs:**
- `session:*`: 24h
- `global` preferences: Never
- Project decisions: 90d (refresh if still relevant)

**Pros:**
- Automatic cleanup
- Explicit intent at write time
- Simple implementation

**Cons:**
- Accidental data loss (wrong TTL)
- Background job complexity (monitoring, failures)
- Hard to "resurrect" accidentally expired memories

**Effort:** Low (2-3 days)

### Option B: Archival Tier with Compression

**Approach:** Don't delete old memories, move to cold storage. Recent memories in hot database; old in S3/object store. Query hot first, fall back to cold if needed.

**Implementation:**
```go
// Nightly job archives memories older than 30d
func ArchiveOldMemories() {
    old := SelectOlderThan(30 * 24 * time.Hour)
    for _, m := range old {
        UploadToS3(m)
        DeleteFromDB(m.ID) // or mark archived
    }
}

// Recall checks hot DB first
func Recall(query string) []Memory {
    hot := QueryDB(query)
    if len(hot) < 5 {
        cold := QueryS3(query) // slower, more expensive
        return append(hot, cold...)
    }
    return hot
}
```

**Pros:**
- No data loss
- Hot storage stays performant
- Cost savings (S3 cheaper than PostgreSQL)

**Cons:**
- Complex infrastructure (S3, Athena for queries)
- Cold retrieval latency (seconds)
- Re-hydration complexity

**Effort:** High (1-2 weeks infrastructure)

### Option C: Relevance-Based Graduated Decay

**Approach:** Memories lose "relevance score" over time. Frequently accessed memories stay fresh; untouched memories fade and eventually delete. Organic garbage collection.

**Algorithm:**
```go
func UpdateRelevance(memory Memory) {
    daysSinceAccess := now.Sub(memory.LastAccessed).Hours() / 24
    memory.RelevanceScore *= math.Pow(0.99, daysSinceAccess) // 1% decay per day
    
    if memory.RelevanceScore < 0.1 {
        Delete(memory.ID) // Or archive
    }
}
```

**Boosts:**
- Accessed: +0.1
- Recalled and used: +0.2
- Explicitly upvoted: +0.5

**Pros:**
- Natural selection (useful memories survive)
- No arbitrary TTLs
- Self-tuning to usage patterns

**Cons:**
- Unpredictable (when will my memory disappear?)
- Important but rarely accessed memories may be lost
- Complex tuning (decay rates, boost values)

**Effort:** Medium-High (scoring system + background job)

**Recommendation:** Option A (TTL) is simplest and most predictable. Start with explicit expiration for session contexts. Option C (graduated decay) is elegant but risky—deferred until we have confidence scoring working.

## Problem 6: Cross-System Integration

**Current State:** Trindex is isolated. Other systems (issue trackers, documentation, CI/CD) don't share context. Agents working across systems lose continuity.

**Impact:** Siloed knowledge, repeated explanations, no system-of-record for user context.

### Option A: MCP-First Architecture (Expand Tool Set)

**Approach:** Make Trindex the universal memory layer via MCP. Build adapters for external systems to read/write through Trindex. All agents/tools use Trindex as single source of truth.

**Adapters:**
```
External System → Trindex Adapter → Trindex Core
- Linear issues   → Remember bug reports, recall related decisions
- Notion docs     → Remember architecture, recall during coding
- GitHub PRs      → Remember review patterns, recall for new PRs
- Slack           → Remember decisions, recall context in threads
```

**Implementation:**
- Build MCP servers for each external system
- Map external concepts to Trindex namespaces
- Unified query interface across all data

**Pros:**
- Single memory system (simplicity)
- All context in one place
- Rich cross-system queries

**Cons:**
- Heavy integration burden (every new system needs adapter)
- Potential performance bottleneck
- Single point of failure

**Effort:** High (ongoing integration work)

### Option B: Event-Driven Federation

**Approach:** Decentralized memory. Each system keeps its own store, emits events on changes. Trindex subscribes to relevant events, creates shadow copies for cross-system queries.

**Architecture:**
```
Linear ──┐
Notion ──┼──→ Event Bus (Kafka/NATS) ──→ Trindex (shadow copy)
GitHub ──┘
Slack ────┘
```

**Event Types:**
- `memory.created` {source, namespace, content, metadata}
- `memory.updated` {id, patch}
- `memory.deleted` {id}

**Implementation:**
- Systems emit events (webhook or SDK)
- Trindex consumes, stores in `shadow:{system}` namespace
- Write-through: Changes to shadow propagate back via events

**Pros:**
- Systems remain autonomous
- Loose coupling (systems can change independently)
- Scalable (each system handles own load)

**Cons:**
- Eventual consistency (lag between systems)
- Conflict resolution (what if both systems update?)
- Complex event infrastructure

**Effort:** High (1-2 months for infrastructure)

### Option C: Context Passport Pattern

**Approach:** Agents carry context between systems explicitly. When switching from coding to issue tracking, agent exports relevant Trindex context, imports into target system.

**Implementation:**
```go
// Agent switching from VS Code to Linear
func SwitchContext(fromSystem, toSystem string) ContextPassport {
    relevant := Recall("current work", ["trindex"], 0.5, 10)
    return ContextPassport{
        Source:    fromSystem,
        Target:    toSystem,
        Memories:  relevant,
        Timestamp: now(),
    }
}

// Target system imports
func ImportPassport(passport ContextPassport) {
    for _, m := range passport.Memories {
        LocalStore.Remember(m.Content, metadata={source: passport.Source})
    }
}
```

**Pros:**
- No infrastructure changes
- Systems don't need to know about each other
- Explicit context sharing (privacy control)

**Cons:**
- Manual agent coordination
- Context may be stale by time of import
- Duplication across systems

**Effort:** Low (protocol definition only)

**Recommendation:** Option A (MCP-first) is the endgame but heavy. Start with Option C (passport pattern) for immediate cross-system flow. Build adapters incrementally toward Option A. Defer Option B unless we have 5+ systems to integrate.

**Phase 1 (Immediate - Week 1-2):**
- Implement Problem 1 Option A (client-side dedup)
- Document Problem 2 Option A (namespace convention)
- Enforce Problem 4 Option A (session initialization)

**Phase 2 (Short-term - Month 1):**
- Problem 1 Option B (server-side hash dedup)
- Problem 3 Option A (JSON metadata column)
- Problem 5 Option A (TTL for session contexts)

**Phase 3 (Medium-term - Month 2-3):**
- Problem 2 namespace migration (consolidate `default`)

- Problem 3 Option C (auto-extraction if metadata sparse)
- Problem 5 Option C (graduated decay for long-term storage)
